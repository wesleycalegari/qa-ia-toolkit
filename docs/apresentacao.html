Fluxo Principal

Do Card à Execução e Encerramento

1
Consultar o board da sprint
Query WIQL no Azure DevOps — lista todos os cards da iteração atual com status, tipo (PBI ou Bug) e responsável.
POST /wit/wiql · IterationPath = 'CITWEB\Sprint_N'
2
Ler o card completo
Título, descrição, critérios de aceite, tasks filhas, seguradora e ramo alvo. Consultar também os manuais do sistema (manuais ia\) e as features documentadas (features\) antes de formular qualquer CT.
⏸
Gate 1 — Card incompleto?
Se critérios de aceite ausentes, regra ambígua, ramo ou ambiente não declarado → postar comentário no card com as dúvidas, mencionando @fabiola.costa. Não avançar sem resposta.
3
Mapear a tela via Playwright MCP
Navegar pela UI real — sempre partindo do login, clicando pelo menu, nunca acessando URL direta. Tirar snapshot de todos os campos, expandir abas, executar o fluxo completo uma vez para descobrir modais e comportamentos não óbvios.
Playwright MCP · browser_navigate → browser_click → browser_snapshot
4
Rascunhar os Casos de Teste
Formato CT-XXX-NNN. Cada clique, campo e modal = um passo separado. Passos com dados reais da tela — nunca inventados. Último passo sempre começa com "Verificar que…". Para CNPJ: pares padrão/alfanumérico lado a lado.
⏸
Gate 2 — Revisão individual de cada CT
Apresentar um CT (ou par padrão/alfanumérico) de cada vez ao Wesley. Aguardar aprovação explícita ou correção antes de passar para o próximo. Nunca gravar sem aprovação de todos os CTs do lote.
5
Gravar passos no Excel
Script Python com openpyxl — escreve os passos na coluna H (col 8) da aba correta. Wesley fecha o arquivo antes da execução.
row = CT_number + 12 · wb.save() após todas as células
6
Gerar o HTML do plano de teste
Tema escuro obrigatório. Estrutura: header com badge PBI/Bug, dashboard de métricas com progress bar, CTs em accordion (ct-block/ct-bar/ct-panel), matriz de rastreabilidade e seção de evidências por CT.
Template: sprint-7-citweb/pbi-6841/index.html · Vars: --bg #0f1114, --accent #3b9eff
⏸
Gate 3 — Revisão do plano completo
Wesley revisa o HTML aberto no browser — conteúdo, estrutura, métricas, evidências. Nunca publicar sem aprovação explícita.
7
Publicar no GitHub Pages
Script publicar.py faz push da pasta docs/sprint-N-citweb/<card>/ para o repositório wesleycalegari/planos-de-teste.
URL: https://wesleycalegari.github.io/planos-de-teste/sprint-N-citweb/bug-XXXX/
8
Comentar no Azure DevOps
Comentário HTML rico com link clicável para o plano publicado + menção @fabiola.costa solicitando validação.
POST /wit/workItems/{id}/comments · api-version 7.1-preview.3
⏸
Gate 4 — Validação da Fabíola + PBI liberado para homologação
Aguardar a validação do plano por @fabiola.costa e a transição do card para "Pronto para Homologação" antes de iniciar qualquer execução.
9
Executar o plano de teste
Execução de todos os cenários via Playwright MCP, navegando pelo menu real. Cada CT recebe screenshot de evidência, classificação (aprovado / reprovado / bloqueado) e o resultado é atualizado no HTML do plano em tempo real. Se algum CT reprovar, o sub-fluxo de bug é acionado.
Playwright MCP · browser_snapshot · evidencias/ por CT · updateDash()
10
Comentar no card com resultado completo
Após todos os cenários aprovados: comentário HTML rico no card do AzDO com banner de status, tabela de resultados por CT, evidências vinculadas como attachments e link para o plano publicado.
POST /wit/workItems/{id}/comments · upload evidências via /wit/attachments
11
Notificar o canal Teams
Adaptive Card no canal CITWEB Melhorias via webhook Power Automate. Card com: status final do card, FactSet de resultados por CT, versão do ambiente, botão com link para o plano publicado.
POST webhook Power Automate · content-type: application/vnd.microsoft.card.adaptive
Sub-fluxo

Quando um CT é Reprovado

01
Capturar evidências e contexto completo
Screenshots antes/durante/depois da falha, resposta raw do endpoint e texto exato da mensagem de erro. Registrar também a descrição completa do CT e os critérios de aceite do card — garantindo que o relatório seja autoexplicativo para o desenvolvedor, sem necessidade de consultar o AzDO para entender o que era esperado. Identificar contraprova quando possível.
02
Gerar bug report HTML
Relatório estruturado com: critérios de aceite violados, comportamento obtido vs esperado, tabela de passos com a linha de falha destacada em vermelho, hipótese de causa raiz e evidências inline. O relatório deve ser suficiente para que o dev reproduza e corrija sem precisar de explicações adicionais.
⏸
Gate — Confirmação obrigatória
Apresentar o bug report ao Wesley e aguardar confirmação explícita ("é bug", "confirmo", "pode abrir") antes de criar qualquer item no AzDO.
03
Consultar PBI pai
Copiar AreaPath e IterationPath exatos do card pai via API — nunca usar "CITWEB" como raiz.
04
Criar task Bug no AzDO
Tipo Task, título [BUG] CT-XXX — descrição, link Hierarchy-Reverse para o PBI, tag rotina.
05
Anexar evidências
Upload via /wit/attachments (octet-stream) → vincular URL retornada como AttachedFile na task.
06
Comentar e notificar
Na task: mencionar o dev assignado. No PBI: mencionar @fabiola.costa informando o registro do bug.

Aprendizados

Dificuldades e Como Foram Resolvidas

01
Playwright local não funciona no Windows
Scripts Python com Chromium bundled falhavam com timeout de 180s — problema de isolamento do ambiente Windows com EDR/XDR corporativo.
Resolvido: migração total para Playwright MCP. Nunca mais script local.
02
Falsos positivos na classificação de CTs
Sistema bloqueava por razão errada — ex: duplicidade de data mascarava o erro de vigência que o CT realmente testava (BUG-6717, CT-CCV-006). CT era marcado como PASSOU, mas pelo motivo errado.
Resolvido: limpeza obrigatória de grade antes de cada CT de bloqueio + verificação explícita do texto da mensagem de erro antes de classificar.
03
PowerShell bloqueado pelo EDR/XDR corporativo
Comandos como IEX, EncodedCommand e execução de Base64 são bloqueados pelo antivírus corporativo. Scripts que pareciam simples falhavam silenciosamente.
Resolvido: código limpo, auditável, com comentários e sem ofuscação. -ExecutionPolicy Unrestricted em vez de Bypass. Nunca mais 1s entre comandos.
04
Massa de dados incompleta bloqueando CTs
CTs paravam por falta de apólice, CC, subgrupo ou usuário. Aguardar o time de infra atrasava dias inteiros de execução.
Resolvido: criar a massa diretamente via banco HML (query SQL) ou pela própria UI do CITWEB, sem depender de terceiros.
05
AzDO sem permissão de Test Management
PAT sem acesso à área de Test Plans — impossível criar suítes e test cases via API no Azure DevOps Test Management.
Alternativa validada: planos HTML no GitHub Pages com dashboards interativos, rastreados por comentário no card AzDO. Solução mais rica visualmente que os test plans nativos.
06
Card criado no board errado no AzDO
Em 15-16/06/2026 um card Bug foi criado com AreaPath = "CITWEB" (raiz) em vez do AreaPath do PBI pai — ficou no board errado, invisível para o time.
Resolvido: antes de criar qualquer card filho, consultar o PBI pai via API e copiar o AreaPath e IterationPath exatos. Regra gravada como bloqueante no processo.
07
Escala do plano CNPJ alfanumérico
182 CTs cobrindo mais de 40 telas diferentes — módulo por módulo, cada campo com par padrão e alfanumérico. Volume inviável de escrever manualmente.
Solução: mapeamento de tela via MCP antes de cada bloco → rascunho assistido → revisão individual → scripts de gravação incremental por lote. Processo iterativo sem acúmulo.
08
Navegação em telas com iframes e comportamentos dinâmicos
Formulários dentro de iframes (CondicoesComerciais59.aspx), botões que são <a> em vez de <button>, postbacks após select_option(), modais Bootstrap invisíveis ao click normal do Playwright.
Resolvido: mapeamento de cada tela antes de escrever os CTs. Seletores exatos documentados por módulo. Padrões específicos registrados na memória e reutilizados.

Visão geral

O que é este setup

Claude Code atua como um par de QA que lê cards, navega na aplicação real, escreve casos de teste, executa via Playwright e gera todos os artefatos — mas nunca age sozinho em pontos críticos. Gates de aprovação garantem que o QA humano valida antes de qualquer publicação no Azure DevOps ou notificação ao time.
O que ele faz
  • Lê e interpreta cards do Azure DevOps
  • Navega na aplicação via Playwright MCP
  • Escreve CTs com passos precisos e reais
  • Gera planos HTML interativos
  • Executa testes e captura evidências
  • Publica resultados e notifica o time
O que ele NÃO é
  • Não substitui o julgamento do QA
  • Não publica sem aprovação do responsável
  • Não inventa dados — confirma antes de assumir
  • Não cria bugs no AzDO sem confirmação humana
  • Não sabe regras de negócio que não foram ensinadas

Skills

O que são e como usar

Skills são arquivos .md que ensinam ao Claude um conjunto de regras, formatos e comportamentos para uma atividade específica. Ficam em ~/.claude/commands/ e podem ser invocadas com /nome-da-skill ou ativadas automaticamente quando o contexto corresponde.

qa-expert
QA Expert — Planejamento
Define o formato completo de um plano de teste: seções obrigatórias, convenção de IDs, prioridades, grupos, subgrupos, critérios de aprovação, riscos. É a fonte de verdade para qualquer plano gerado.
Quando usar: ao criar plano de teste, analisar PBI/Bug, definir escopo e CTs de um card.
validacao-testes
Validação de Testes — Execução
Define o fluxo de execução: como navegar na aplicação, quando tirar screenshot, como classificar cada resultado (aprovado/reprovado/bloqueado), o que fazer ao encontrar um CT reprovado, formato das evidências.
Quando usar: ao executar CTs, capturar evidências, registrar resultados, gerar bug report.
Skill de domínio
Skill específica do seu projeto
Você cria uma skill com as regras de negócio, fluxos e nomenclaturas do seu sistema. No CITWEB/CITNET chamamos de citweb-expert. No seu projeto pode ser meu-sistema-expert. Essa skill descreve o sistema, suas telas, seus módulos e regras de negócio relevantes para o QA.
Quando usar: sempre que o Claude precisar entender o contexto do seu sistema para escrever CTs corretos.
Dica — Deixe o Claude aprender sozinho

Se o seu projeto tiver código-fonte, manuais, documentação ou arquivos de features disponíveis, você não precisa descrever tudo manualmente na skill. Basta instruir o Claude a ler esses materiais antes de agir — ele extrai as regras de negócio, fluxos e comportamentos sozinho. Quanto mais material disponível, mais precisa é a geração dos CTs, sem que o QA precise documentar o sistema.

No CLAUDE.md do projeto
# Fontes de verdade — ler antes de agir - Manuais: manuais/ (Markdown ou PDF) - Código: src/ (regras de negócio) - Features: features/ (comportamentos esperados) - Swagger: docs/api/ (contratos de API)
Na skill de domínio
# Instrução na skill ## Antes de gerar qualquer CT 1. Ler manuais/ — entender fluxos 2. Ler features/ — comportamentos 3. Grep em src/ — confirmar regras 4. Só então rascunhar passos
Código-fonte disponível
Claude lê os controllers, services e validators para entender as regras sem que o QA precise explicá-las.
Manuais em Markdown ou PDF
Claude extrai fluxos, campos obrigatórios e exceções diretamente da documentação do sistema.
Swagger / OpenAPI
Claude lê os contratos de API para gerar CTs de integração com campos, tipos e respostas corretas.
Features / BDD
Arquivos .feature com cenários Gherkin são a fonte mais direta — Claude converte em CTs estruturados automaticamente.
Configuração

Adaptando ao seu projeto em 5 passos

1
Crie o arquivo CLAUDE.md do projeto
O CLAUDE.md é lido automaticamente pelo Claude a cada conversa. Nele você coloca as regras que se aplicam sempre: qual sistema é, qual o ambiente de HML, como navegar no sistema, quais manuais consultar antes de agir.
# CLAUDE.md — MeuProjeto ## Regras sempre ativas - Playwright MCP sempre — nunca scripts locais - Navegar por clique — login → menu → tela (nunca URL direta) - Evidências obrigatórias — screenshot em toda ação relevante - features/ é a fonte de verdade — ler antes de documentar fluxo - Nunca publicar sem revisão — apresentar ao responsável antes de push ## Auto-aplicação de skills # Planejamento de testes Ler: ~/.claude/commands/qa-expert.md # Execução e evidências Ler: ~/.claude/commands/validacao-testes.md
2
Configure o Playwright MCP no .mcp.json
Crie ou edite o arquivo .mcp.json na raiz do projeto com a configuração do servidor Playwright. Isso permite ao Claude abrir o browser, clicar, preencher campos e tirar screenshots da aplicação real.
{ "mcpServers": { "playwright": { "command": "npx", "args": ["@playwright/mcp@latest"] } } }
3
Salve as credenciais no .env
Crie um arquivo .env na raiz com o PAT do AzDO e o token do GitHub. O Claude lê este arquivo para fazer chamadas de API sem expor credenciais no código. Nunca commite o .env — adicione ao .gitignore.
# .env AZDO_PAT=seu_personal_access_token_aqui GIT_PAT=seu_github_token_aqui AZDO_ORG=SuaOrganizacao AZDO_PROJECT=SeuProjeto
4
Crie a skill de domínio do seu sistema
Crie o arquivo ~/.claude/commands/meu-sistema-expert.md ensinando ao Claude o que é o seu sistema: telas principais, módulos, fluxos típicos, regras de negócio críticas e nomenclaturas específicas. Se o projeto tiver código-fonte, manuais ou documentação acessíveis, instrua o Claude a lê-los diretamente — ele aprende as regras sem que o QA precise documentar tudo na skill. A skill pode ser mínima e apontar para os arquivos que já existem.
## Sistema Nome: MeuSistema Tipo: ERP / Portal / API / ... Stack: React + .NET / Laravel / ... ## Ambientes HML: https://hml.meusistema.com Credenciais: ver .env ## Módulos principais - Cadastro de Clientes: fluxo CRUD completo - Pedidos: criação → aprovação → entrega - Relatórios: exportação PDF e Excel ## Regras de negócio críticas - Pedido só pode ser aprovado se cliente tiver crédito disponível - Data de entrega não pode ser retroativa - # adicione as regras relevantes para o QA ## Convenção de IDs dos CTs CT-CAD-001 → Cadastro CT-PED-001 → Pedidos CT-REL-001 → Relatórios
5
Configure a pasta de planos e o script de publicação
Crie a estrutura de pastas para os planos e copie o script publicar.py que faz push para o GitHub Pages. Adapte apenas as variáveis de repositório e sprint. A estrutura de pastas é o único ponto que precisa ser ajustado por projeto.
planos-de-teste/ docs/ sprint-1-meuprojeto/ bug-001/ index.html ← plano de teste HTML evidencias/ CT-CAD-001/ ← screenshots por CT CT-PED-002/ sprint-2-meuprojeto/ pbi-050/ index.html

Estrutura de pastas sugerida

Catálogo de scripts — obrigatório manter

O Claude mantém um arquivo _catalogo_scripts.md com um índice de todos os scripts existentes — o que cada um cobre, quando foi criado e em quais sprints passou. Antes de gerar um novo script, o Claude consulta esse catálogo para evitar duplicação e reaproveitar o que já existe.

## Catálogo de Scripts de Automação ct001_cadastro_cliente.py Cobre: Cadastro > Cliente > CRUD completo Sprint: 1 (PASS), 2 (PASS) → promovido ao regressivo Reusar quando: qualquer CT de cadastro de cliente ct005_aprovacao_pedido.py Cobre: Pedidos > Aprovação > fluxo gerente Sprint: 2 (PASS), 3 (PASS), 4 (PASS) Reusar quando: reteste de aprovação ou permissões de pedido
Regra de promoção para o regressivo: um script é movido de _scripts/ para _regressivo/ quando passa sem alteração em 2 ou mais sprints consecutivas. A partir daí, é executado automaticamente em toda nova release para garantir que nenhuma entrega quebra o que já funcionava. O orquestrador rodar_regressivo_completo.py executa todos e gera um dashboard HTML com os resultados.
Princípios

Convenções que funcionam em qualquer projeto

Estes princípios foram validados ao longo de 9 sprints e se aplicam independentemente do sistema, da linguagem ou da stack testada.

🖱️
Sempre navegar por clique
Nunca usar URL direta para chegar a uma tela. Sempre partir do login e navegar pelo menu como um usuário real faria.
Por quê: URLs internas mudam. O menu é o contrato real de navegação do sistema.
📸
Screenshot em toda ação relevante
A evidência é obrigatória. Cada CT aprovado ou reprovado precisa de screenshot que mostre claramente o resultado — especialmente a mensagem de sucesso ou erro visível na tela.
Por quê: sem evidência, o resultado não pode ser auditado nem reproduzido.
🔍
Ler a mensagem de erro, não só o comportamento
Antes de classificar um CT como PASSOU ou FALHOU, verificar qual mensagem exata o sistema retornou. O sistema pode bloquear pela razão errada e o CT passar por motivo incorreto.
Por quê: falsos positivos são mais perigosos que falhas óbvias.
⏸️
Gates de aprovação antes de publicar
Nunca publicar nada no AzDO (plano, bug, comentário) sem aprovação explícita do responsável. Apresentar sempre antes de agir.
Por quê: publicar informação errada no card do time tem custo alto. Reverter é trabalhoso.
🗺️
Mapear a tela antes de escrever o CT
Antes de rascunhar qualquer caso de teste, navegar até a tela real via Playwright, tirar snapshot e executar o fluxo pelo menos uma vez. Só então escrever os passos.
Por quê: CTs escritos sem ver a tela têm passos incorretos, labels errados e modais ignorados.
🧹
Estado limpo antes de cada CT
Cada CT começa em estado limpo — sem dados residuais de execuções anteriores que possam interferir no resultado. Se necessário, incluir um passo explícito de limpeza.
Por quê: dados anteriores podem fazer o sistema bloquear por razão diferente do que o CT testa.
❓
Nunca deduzir — sempre confirmar
Se qualquer dado estiver faltando (seguradora, número de registro, valor, ambiente), perguntar antes de prosseguir. Jamais preencher com suposição.
Por quê: planos com dados deduzidos precisam ser refeitos. O custo de perguntar é zero.
🐛
Bug confirmado pelo humano antes do AzDO
Ao encontrar CT reprovado, gerar o bug report HTML e aguardar confirmação explícita do responsável antes de criar qualquer item no AzDO.
Por quê: evita registrar falsos positivos como bugs reais no board do time.

Antes de começar o primeiro ciclo de QA em um projeto novo, verifique estes itens.

Claude Code instalado e autenticado com sua conta Anthropic
Playwright MCP configurado no .mcp.json e testado (abrir uma URL)
PAT do Azure DevOps criado com permissões de Work Items (read + write) e Attachments
Arquivo .env criado com as credenciais e adicionado ao .gitignore
CLAUDE.md criado na raiz do projeto com as regras do sistema
Skill de domínio criada em ~/.claude/commands/meu-sistema-expert.md
Repositório GitHub configurado com GitHub Pages habilitado
Script publicar.py adaptado com o repositório e estrutura de pastas do projeto
Estrutura de pastas planos-de-teste/docs/sprint-N/ criada
Acesso ao ambiente HML do sistema confirmado (URL + credenciais de teste)
Template HTML do plano de teste adaptado com as cores e nome do projeto
Primeiro CT escrito, revisado e aprovado pelo responsável antes de publicar qualquer coisa
Dica para a primeira sessão: peça ao Claude para ler o card do AzDO, mapear a tela principal via Playwright e apresentar apenas 1 ou 2 CTs para validação antes de qualquer automação. Isso calibra o formato e as expectativas do time sem investimento alto.
